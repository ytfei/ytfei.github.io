代码评审

Posted on 2019-12-18 In 开发 , 清单

清单：代码评审

开发人员代码提交评审前，请参考该规范提前自查。

准备

代码提交评审前，请先适当整理：

  1. 代码规范请参考 阿里巴巴Java开发.pdf
  2. 格式化代码（保持代码风格统一）
  3. 使用 IDEA Alibaba Coding Guidelines Analyze 工具对代码进行扫描，并对发现的问题进行处理

评审

评审的内容除了参考执行 阿里巴巴Java开发.pdf 以外，还需要从以下几个方面考虑：

  1. 模块设计
    1. 功能性
      1. 描述功能被使用的角色&场景
      2. 描述功能要解决的主要问题
      3. 业务流程说明
      4. 数据库表结构及核心业务状态
    2. 非功能性
      1. 性能：场景对性能（时延、吞吐量）的要求
      2. 扩展性：场景是否存在多个备选项
      3. 幂等性：重复性操作是如何处理
      4. 安全性：
      5. 权限、数据隐私（是否脱敏或者加密）
      6. 三方资源（付费，如短信等）保护，防止被刷
  2. 代码实现
    1. 检查是否遵守了 阿里巴巴Java开发.pdf
    2. 代码方案实现的整体模式
    3. 代码注释、变量、函数命名是否清晰易懂
    4. 代码日志能否清晰描述业务过程
      1. 日志级别是否正确
      2. 日志内容要恰当，既不能模糊，也不能事无巨细的全部写在日志
      3. 日志中不能包含隐私内容
    5. 异常处理过程
      1. 不允许捕获异常而不做处理（捕获异常记录日志，也是一种不处理的情况，要注释说明原因）
      2. 捕获异常是否会影响正常的流程
      3. 捕获异常时，是否需要重试，或者检查业务状态？
    6. 多线程
      1. 是否使用线程池，线程池的生命周期，队列的长度，现成的数量
      2. 线程任务是否有依赖性，线程任务的执行时间是否会过程
      3. 线程池能否关闭、何时关闭
    7. 锁（本地锁和分布式锁）
      1. 为什么需要使用锁，说明原因
      2. 锁的获取与释放
      3. 锁超时的设置
      4. 任务之间是否存在死锁的可能性
      5. 锁是否可能成为全局的瓶颈
      6. 有没有不用锁的替代方案
    8. 数据库事务
      1. 说明使用数据库事务的原因
      2. 事务失败会产生什么影响
      3. 事务是否需要分拆，避免事务过大
      4. 是否对事务的有效性做过验证（声明式事务配置不当可能不会生效）
  3. 升级部署
    1. 兼容性
      1. 是否有数据库变更
      2. 是否有配置文件变更
      3. 是否需要停服发布
    2. 能否回滚
      1. 是否有回滚预案，风险是什么
